# イベントストーミング

ドメインを分析し、戦略/戦術設計の土台となる成果物を `docs/` に残すための手順。**分析を最優先**し、結論(集約/BC/コア)に飛びつかず、イベントの洗い出しから積み上げる。

## 進め方(対話的に、一段ずつユーザーと合意する)

### 1. Big Picture（全体像）
- ドメインで起きる**ドメインイベント(過去形)**を時系列で洗い出す(例: `在庫が入荷された`, `在庫が引き当てられた`, `出荷された`)。
- ホットスポット(疑問・対立・曖昧な点)を付箋として残す。
- 成果物: `docs/event-storming/01-big-picture.md`(イベント時系列 + ホットスポット一覧)。

### 2. Process Modeling（プロセス）
- 各イベントについて、それを引き起こす**コマンド(命令形)**、実行する**アクター/ロール**、判断を担う**ポリシー**、参照する**リードモデル**を紐づける。
- 外部システム連携点を明示する(例: 上流の受発注は外部トリガとして薄く扱う)。
- 成果物: `docs/event-storming/02-process.md`(コマンド→イベントの流れ + ポリシー)。

### 3. Software Design（設計への橋渡し）
- 強い整合性が必要なコマンド/イベントの塊から**集約候補**を見出す(不変条件で括る)。
- 言語の境界・変化の理由の違いから**境界づけられたコンテキスト(BC)**を切り、**コンテキストマップ**を描く。
- 事業の差別化・難易度から**コアサブドメイン**を特定し、支援/汎用サブドメインと区別する。
- **ユビキタス言語表**を作る(用語・意味・英名)。
- 成果物: `docs/event-storming/03-software-design.md`、`docs/context-map.md`、`docs/ubiquitous-language.md`。

## 記法
- 図は Mermaid で書く(`flowchart` でコマンド→イベント、`graph` でコンテキストマップ)。
- 命名は `.claude/rules/ddd-ubiquitous-language.md` に従う(コマンド=命令形、イベント=過去形、集約=名詞)。

## 完了条件
- BC一覧・コンテキストマップ・コアサブドメイン・集約候補・ユビキタス言語表が `docs/` に揃い、次工程(戦術設計/実装)の入力として使える状態になっていること。
